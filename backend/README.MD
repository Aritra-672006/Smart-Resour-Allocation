## ЁЯМР API Reference

### `GET /health`
Health check endpoint.

```json
{ "status": "ok", "service": "Smart Resource Allocation API", "timestamp": "..." }
```

---

### `POST /upload-report`
Submit an NGO field report for AI processing.

**Request body:**
```json
{
  "ngo_name": "ABC NGO",
  "location": "Kolkata",
  "report_text": "Flood situation тАФ 500 families displaced",
  "people_affected": 500
}
```

**Flow:** Save тЖТ Fetch volunteers тЖТ Call ML `/ai-agent` тЖТ Store `ai_result` тЖТ Create assignments тЖТ Respond

**Response `200`:**
```json
{
  "message": "Report processed successfully",
  "report_id": "<ObjectId>",
  "ai_result": { ... }
}
```

**Errors:** `400 validation_error` | `500 internal_server_error`

---

### `POST /register-volunteer`

**Request body:**

## ЁЯдЦ ML Service Integration

The ML microservice is called at `POST {ML_BASE_URL}/ai-agent` with:

```json
{
  "text": "<report_text>",
  "volunteers": [ ...sortedByScore ]
}
```

Expected response shape (fields used by backend):

```json
{
  "need_analysis": [
    { "need_type": "Food", "location": "...", "urgency": "high", "people_affected": 200 }
  ],
  "tasks": [
    { "title": "...", "location": "...", "urgency": "...", "assigned_volunteers": [] }
  ],
  "decision_trace": { ... }
}
```

If the ML service is unavailable, the report is still saved with a degraded `ai_result` and a single fallback assignment is created тАФ the system **never hard-fails** on ML errors.
